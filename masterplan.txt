Build order
Phase 1 — Get a usable local chat stack running ✅ COMPLETE
Goal: prove the model, app shell, and local API all work before touching ingestion.
Step 1. Set up the local runtime ✅
Used:

Ollama (installed on Windows)
Gemma 4 (pulled via ollama pull gemma4)
Open WebUI was not used

Deliverable:
Model responds to API calls at http://localhost:11434/api/chat. Verified via PowerShell using Invoke-WebRequest.
Notes:

PowerShell aliases curl to Invoke-WebRequest, so standard curl syntax doesn't work. Use Invoke-WebRequest with -Method POST, -ContentType, and -Body parameters instead.
PowerShell may show a "Security Warning: Script Execution Risk" prompt when parsing web responses. Answering Y or adding -UseBasicParsing to the command avoids this in the future.
Gemma 4 was used instead of the originally planned Gemma 3. Update downstream references accordingly.

Phase 2 — Create the app shell ✅ COMPLETE

Goal: make the interface you will actually use from desktop and phone.

Step 2. Build the Next.js app ✅

Used:

Next.js 16.2.3 (React 19, App Router, TypeScript)
No auth — local-only for now, can add later
Tailwind 4

Core pages built:

/chat          — conversation index ("select or start a chat")
/chat/[id]     — individual conversation with streaming chat UI
/upload        — placeholder for Phase 3 ingestion
/library       — placeholder for parsed file browser
/wiki          — placeholder for wiki page viewer
/settings      — shows current runtime config (Ollama URL, model, storage paths)

Layout:

AppShell wraps all pages with a persistent sidebar and mobile hamburger menu
Sidebar shows primary nav + live conversation list with delete
/ redirects to /chat

Deliverable:

Mobile-friendly app shell with sidebar nav and working chat UI.

Notes:

No Prisma used — raw better-sqlite3 is simpler and sufficient.
Auth was deferred — this is a local-only tool for now. Add if/when exposing externally.
The /upload, /library, /wiki pages are intentional placeholders — they show clear
  "coming in Phase N" messaging so the nav is fully wired before the features exist.

Step 3. Add chat persistence ✅

Used:

SQLite via better-sqlite3
Database at storage/eidetic.db
No ORM — raw prepared statements

Stored:

conversations (id, title, mode, created_at, updated_at)
messages (id, conversation_id, role, content, sources_json, created_at)

API routes:

GET  /api/conversations            — list all, ordered by updated_at desc
POST /api/conversations            — create new conversation
GET  /api/conversations/[id]       — fetch conversation with full message history
PATCH /api/conversations/[id]      — update title
DELETE /api/conversations/[id]     — delete conversation and cascade messages
POST /api/conversations/[id]/messages — send user message, stream Ollama response,
                                         persist both turns; auto-titles from first message

Streaming:

Ollama /api/chat called server-side with stream: true
Response piped token-by-token to the client as plain text
Full assistant response persisted to DB once stream closes

Deliverable:

Chats persist across sessions. Conversation list is live in sidebar.
Streaming responses feel instant. Build passes clean (npm run build verified).

Notes:

Model name in Ollama is gemma3 (not gemma4 — that was the product name, not the pull ID).
Override with env vars: OLLAMA_URL, OLLAMA_MODEL.
sources_json column is reserved for Phase 5 RAG — not populated yet.
Auto-title takes the first 60 chars of the first user message; can be improved later.
Phase 3 — Build the raw file ingestion layer ✅ COMPLETE

Goal: turn raw files into normalized text + metadata.

Step 4. Create a file intake workflow ✅

Used:

Next.js multipart upload route
Local file storage under storage/raw/{docId}/

Supported file types:

pdf, docx
pptx
xlsx
receipt images (png, jpg, jpeg)

Raw folders created automatically on startup:

storage/raw
storage/processed
storage/failed

API routes:

POST /api/upload              — multipart upload; saves file, creates DB record (status: queued)
GET  /api/documents           — list all documents
GET  /api/documents/[id]      — fetch document + fragments
DELETE /api/documents/[id]    — delete file + DB record
POST /api/documents/[id]/parse — spawn Python parser detached; marks doc as processing

Deliverable:

Files land in storage/raw and are tracked in the database.

Step 5. Parse documents and slides ✅

Used:

unstructured[pdf,docx] for PDFs and Word documents
python-pptx for PowerPoint

Extracted:

headings, paragraphs, page numbers (PDF/DOCX via unstructured element types)
slide text, slide titles, slide numbers, speaker notes (PPTX via python-pptx)

Deliverable:

Docs and slides become text fragments with metadata stored in document_fragments.

Step 6. Parse spreadsheets properly ✅

Used:

openpyxl (read-only, data-only mode)

Extracted:

sheet name, row number, column headers (first non-empty row), row values
Each row → one fragment; not flattened into a blob
Row values serialized to metadata_json for downstream use

Deliverable:

Spreadsheet rows are individually searchable structured records.

Step 7. Parse receipts ✅

Used:

pytesseract + Pillow (greyscale + sharpen preprocessing)
Heuristic regex for total, subtotal, tax, date extraction
Vendor inferred from first line of OCR output

Extracted:

raw OCR text (fragment_type: ocr_text)
structured fields as a second fragment (fragment_type: receipt_fields): vendor, receipt_date, total, subtotal, tax

Deliverable:

Receipt images become raw text plus structured metadata.

Notes:

Parser is ingestion/parse.py — unified dispatcher by file_type from DB.
Document statuses: queued → processing → processed | failed.
Override Python executable with PYTHON_CMD env var.
Build verified clean.

Phase 4 — Normalize everything into one canonical format ✅ COMPLETE

Goal: make every parser output the same basic record shape.

Step 8. Define the normalized fragment schema ✅

Used:

SQLite via better-sqlite3 (no ORM)

Schema implemented:

documents table — id, original_name, file_type, status, file_path, file_size,
                   fragment_count, error_message, embed_status, embed_error,
                   chunk_count, created_at, updated_at

document_fragments table — id, document_id, text, fragment_type,
                            page_number, slide_number, slide_title,
                            sheet_name, row_number,
                            vendor, receipt_date, total,
                            metadata_json, created_at

Deliverable:

Every file type produces a predictable set of document_fragments rows.
All downstream RAG and wiki work reads from this table.

Notes:

This is the most important backend decision — everything downstream depends on it.

Phase 5 — Build the RAG layer ✅ COMPLETE

Goal: create the ground-truth retrieval system.

Step 9. Chunk the parsed fragments ✅

Approach: one chunk per fragment (no sub-chunking in v1).
Each fragment is already a meaningful unit:
  docs: one chunk per paragraph/heading/element
  slides: one chunk per text block or notes block
  spreadsheets: one chunk per row
  receipts: one chunk for raw OCR text, one for extracted fields

Deliverable:

chunks table in SQLite tracks which fragments have been embedded.
Each chunk is traceable back to its source fragment and document.

Step 10. Generate embeddings ✅

Used:

Ollama /api/embeddings endpoint
Model: nomic-embed-text (override with EMBED_MODEL env var)
Pre-requisite: ollama pull nomic-embed-text

Deliverable:

Every chunk has an embedding vector stored in Chroma.

Step 11. Store vectors and metadata ✅

Used:

chromadb PersistentClient at storage/chroma/
Collection: eidetic_chunks (cosine similarity space)

Stored per chunk:

chunk text (documents)
embedding vector (embeddings)
metadata: document_id, fragment_id, fragment_type,
          page/slide/sheet/row, vendor, receipt_date, total

SQLite chunks table mirrors Chroma IDs for integrity.

Implementation:

ingestion/embed.py — args: <doc_id> <db_path> [chroma_path] [ollama_url]
  Reads fragments → embeds each → upserts to Chroma + SQLite
  Re-embed deletes stale chunks first
  Updates documents.embed_status: null → embedding → embedded | embed_failed

POST /api/documents/[id]/embed — spawns embed.py detached

Deliverable:

Working semantic retrieval over all parsed data.

Step 12. Build a simple retrieval test harness ✅

Used:

ingestion/retrieve.py — Python script called synchronously by the API route
POST /api/retrieve — body: {query: string, n?: number (default 5, max 20)}
                     returns: JSON array of chunks with score + full source metadata

Each result includes:

chunk_id, text, score (cosine similarity 0–1)
document_id, original_name, file_type
fragment_type, page_number, slide_number, slide_title
sheet_name, row_number, vendor, receipt_date, total

Library UI:

Embed / Re-embed / Retry embed button (appears after parsing)
Embed status badges: Embedding… / Embedded / Embed failed
Shows fragment count + chunk count when embedded
Polls every 3s while embedding is in progress

Deliverable:

POST /api/retrieve can verify retrieval quality before wiring into chat.
Phase 6 — Build the LLM Wiki layer ✅ COMPLETE

Goal: create synthesized memory from the parsed/RAG data.

Step 13. Set up the wiki store ✅

Used:

plain Markdown files in storage/wiki/{page_type}/{slug}.md
wiki_pages table in SQLite (id, slug, page_type, title, content, file_path,
  source_doc_ids, dirty, created_at, updated_at)
storage/wiki/vendor/, storage/wiki/doc/, storage/wiki/presentation/, storage/wiki/data/

Page types produced:

vendor        — one page per unique vendor (aggregates all receipt fragments)
doc           — one page per PDF / DOCX document
presentation  — one page per PPTX file
data          — one page per XLSX spreadsheet

Deliverable:

Durable markdown knowledge layer. Files are human-readable and Obsidian-compatible.

Step 14. Build the wiki generator ✅

Used:

Gemma via Ollama /api/chat (non-streaming)
ingestion/wiki.py — unified dispatcher by file_type from DB

Input comes from:

normalized document_fragments (text, page_number, slide_number, slide_title,
  sheet_name, row_number, vendor, receipt_date, total)

Not directly from binary files.

Wiki page responsibilities implemented:

summarize recurring facts
aggregate receipts by vendor across multiple documents
link to supporting sources via Evidence section
preserve uncertainty — system prompt instructs model to say so when evidence is weak

Implementation:

ingestion/wiki.py — args: <doc_id|all> <db_path> [wiki_path] [ollama_url]
  Reads fragments → builds context → calls Ollama → writes .md file → upserts wiki_pages
  Vendor pages aggregate ALL parsed receipts from the same vendor, not just one doc
  Re-generating updates existing wiki_pages row in-place (idempotent)
  Updates documents.wiki_status: null → generating → generated | wiki_failed

POST /api/documents/[id]/wiki — spawns wiki.py detached

Deliverable:

Wiki is generated from grounded data, not chat history.

Step 15. Add source references to wiki pages ✅

Every wiki page includes an ## Evidence section with:

vendor pages:   - receipt: {filename}  (one per contributing receipt)
doc pages:      - doc: {filename}
presentation:   - presentation: {filename}
spreadsheet:    - spreadsheet: {filename}

source_doc_ids column in wiki_pages tracks which documents contributed.

Deliverable:

Every wiki page is auditable and links back to raw source files.

Step 16. Open and inspect the wiki in Obsidian ✅

Wiki files written to storage/wiki/ in standard Markdown — open the folder as
an Obsidian vault directly for page browsing, backlinks, and graph view.

In-app wiki UI also built:

GET /api/wiki          — list all wiki pages (grouped by type in the UI)
GET /api/wiki/[slug]   — fetch a single page with content
app/wiki/page.tsx      — wiki index, grouped by page type with badges
app/wiki/[slug]/page.tsx — individual page viewer (markdown in <pre>, Obsidian path shown)

Library UI:

Generate Wiki / Regenerate Wiki / Retry Wiki button (appears after parsing)
Wiki status badges: Generating… / Wiki / Wiki failed
Wiki error displayed inline
Polls every 3s while wiki is being generated

Notes:

Context passed to the LLM is capped at ~6000 chars to stay within Gemma's context window.
Override Python executable with PYTHON_CMD env var.
Override Ollama URL with OLLAMA_URL env var.
Override model with OLLAMA_MODEL env var (default: gemma3).
Build verified clean.

Phase 7 — Build the hybrid query router

Goal: decide when to use raw RAG, wiki, or both.

Step 17. Create a retrieval orchestration API

Use:

Next.js route handlers or a small backend service

Responsibilities:

classify the question
choose rag, wiki, or hybrid
gather supporting context
build the final prompt
call Ollama
stream the result back

Suggested query modes:

lookup
summary
comparison
trend
citation_required

Routing:

lookup → RAG
summary → wiki first
comparison/trend → hybrid
citation_required → RAG mandatory

Deliverable:

one chat endpoint that intelligently decides how to answer.
Step 18. Build hybrid context assembly

For hybrid mode:

pull top wiki pages
pull supporting raw chunks
send both to the LLM
instruct the model to trust raw evidence over synthesized pages

System rules:

raw source is authoritative
wiki is a convenience layer
cite raw evidence when possible
do not invent support

Deliverable:

better summaries without losing grounding.

One-page summary
What you are building

A local AI knowledge system where:

files are parsed into structured fragments
fragments are indexed in a RAG database
the RAG data generates a persistent markdown wiki
a Next.js app routes user questions to raw retrieval, synthesized memory, or both
the whole system can be accessed from desktop or phone
Best tool choices
Next.js for the app
Ollama for local model runtime
Chroma for vector storage
Unstructured for docs and PowerPoints
openpyxl for Excel
Tesseract/pytesseract for receipt OCR
Markdown + Obsidian for the wiki layer
Tailscale only if you want easy private remote access
