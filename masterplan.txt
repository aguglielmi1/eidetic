Phase 1 — Get a usable local chat stack running ✅ COMPLETE
Goal: prove the model, app shell, and local API all work before touching ingestion.
Step 1. Set up the local runtime ✅
Used:

Ollama (installed on Windows)
Gemma 4 (pulled via ollama pull gemma4)
Open WebUI was not used

Deliverable:
Model responds to API calls at http://localhost:11434/api/chat. Verified via PowerShell using Invoke-WebRequest.
Notes:

PowerShell aliases curl to Invoke-WebRequest, so standard curl syntax doesn't work. Use Invoke-WebRequest with -Method POST, -ContentType, and -Body parameters instead.
PowerShell may show a "Security Warning: Script Execution Risk" prompt when parsing web responses. Answering Y or adding -UseBasicParsing to the command avoids this in the future.
Gemma 4 was used instead of the originally planned Gemma 3. Update downstream references accordingly.

Phase 5 — Build the RAG layer ✅ COMPLETE

Goal: create the ground-truth retrieval system.

Step 9. Chunk the parsed fragments ✅

Approach: one chunk per fragment (no sub-chunking in v1).
Each fragment is already a meaningful unit:
  docs: one chunk per paragraph/heading/element
  slides: one chunk per text block or notes block
  spreadsheets: one chunk per row
  receipts: one chunk for raw OCR text, one for extracted fields

Deliverable:

chunks table in SQLite tracks which fragments have been embedded.
Each chunk is traceable back to its source fragment and document.

Step 10. Generate embeddings ✅

Used:

Ollama /api/embeddings endpoint
Model: nomic-embed-text (override with EMBED_MODEL env var)
Pre-requisite: ollama pull nomic-embed-text

Deliverable:

Every chunk has an embedding vector stored in Chroma.

Step 11. Store vectors and metadata ✅

Used:

chromadb PersistentClient at storage/chroma/
Collection: eidetic_chunks (cosine similarity space)

Stored per chunk:

chunk text (documents)
embedding vector (embeddings)
metadata: document_id, fragment_id, fragment_type,
          page/slide/sheet/row, vendor, receipt_date, total

SQLite chunks table mirrors Chroma IDs for integrity.

Implementation:

ingestion/embed.py — args: <doc_id> <db_path> [chroma_path] [ollama_url]
  Reads fragments → embeds each → upserts to Chroma + SQLite
  Re-embed deletes stale chunks first
  Updates documents.embed_status: null → embedding → embedded | embed_failed

POST /api/documents/[id]/embed — spawns embed.py detached

Deliverable:

Working semantic retrieval over all parsed data.

Each result includes:

chunk_id, text, score (cosine similarity 0–1)

Phase 7 — Build the hybrid query router ✅ COMPLETE

Goal: decide when to use raw RAG, wiki, or both.

Step 17. Create a retrieval orchestration API ✅

Used:

lib/queryRouter.ts — classification + retrieval orchestration
Modified app/api/conversations/[id]/messages/route.ts — inject context before Ollama call

Query categories implemented:

lookup           → RAG   (price, cost, date, how many, receipt, total, amount, invoice)
summary          → wiki first (summarize, overview, what is, tell me about, explain, how does)
comparison       → hybrid (compare, versus, vs, difference between, better than)
trend            → hybrid (trend, over time, historically, pattern, growth, decline)
citation_required→ RAG   (source, proof, evidence, cite, reference, verify)
chat             → no context (everything else — pure conversation)

Routing logic:

lookup / citation_required → RAG only
summary → wiki first; falls back to RAG if no wiki pages match
comparison / trend → hybrid (wiki + RAG together)
chat → passes message history to Ollama with no injected context

Deliverable:

lib/queryRouter.ts exposes routeQuery(query) which classifies the question,
runs retrieval, and returns { mode, category, systemPrompt, chunks, wikiPages, hasContext }.
The messages route prepends a typed system message when context was found.
X-Route-Mode header is returned on every chat response.

Step 18. Build hybrid context assembly ✅

For hybrid mode:

pull top wiki pages (up to 3, SQLite LIKE search on title + content)

send both to Ollama as a system message ahead of chat history
instruct model: raw sources are authoritative; wiki is synthesized memory; cite evidence; do not invent

System rules enforced:

raw source is authoritative
wiki is a convenience layer
cite raw evidence when possible
do not invent support

Context limits:

each wiki page capped at 1 500 chars in the system prompt

Deliverable:

Notes:

No new API routes added — orchestration lives inside the existing messages route.

Step 20. Mark affected wiki pages as dirty ✅

Dirty marking happens at three points:

re-parse (parse.py):  after successful parse, scans wiki_pages.source_doc_ids
                       for the parsed doc_id → sets dirty = 1 + enqueues rewiki job

re-embed (embed.py):  after successful embedding, same dirty marking logic

delete (DELETE /api/documents/[id]):  before deletion, marks wiki pages
                                       referencing the doc as dirty + enqueues rewiki job

Job deduplication: pending rewiki jobs for the same wiki page slug are not duplicated.

Examples:

new Staples receipt → vendor page “Staples” dirty
re-parsed onboarding deck → onboarding doc page dirty
deleted spreadsheet → monthly expense data page dirty

Deliverable:

wiki rebuilds become incremental instead of global.

Step 21. Rebuild only affected wiki pages ✅

Used:

ingestion/wiki.py `dirty` mode — rebuilds only wiki pages marked dirty
POST /api/wiki/rebuild-dirty — spawns wiki.py in dirty mode
GET /api/wiki/rebuild-dirty — returns dirty page count + pending jobs

Rebuild logic:

find all wiki_pages with dirty = 1
for each dirty page, gather valid source documents
if all sources deleted → remove orphaned wiki page + markdown file
otherwise → re-run process_document for the first source doc
  (vendor pages automatically aggregate all docs for that vendor)
job_queue entries updated: pending → running → completed | failed

Library UI:

amber banner shows count of out-of-date wiki pages
“Rebuild Dirty Pages” button triggers incremental rebuild
polls every 3s while rebuild is in progress
dirty count auto-clears when rebuild finishes

Wiki page index:

“Out of date” badge shown on dirty wiki pages in /wiki index

API routes added:

POST /api/wiki/rebuild-dirty — trigger dirty rebuild
GET  /api/wiki/rebuild-dirty — check dirty status + pending jobs

Notes:

wiki_pages.dirty is set to 0 by upsert_wiki_page (existing behavior from Phase 6).
Build verified clean.

Phase 11 — Secure remote access with Tailscale Funnel + auth ✅ COMPLETE

Goal: access eidetic from anywhere over HTTPS with no VPN client on the phone,
protected by real authentication so the Funnel URL is not an open door.

Implementation note:

The original plan specified NextAuth (Auth.js v5), but Next.js 16 renamed
middleware.ts to proxy.ts and deprecated the middleware convention. NextAuth v5
relies on wrapping the middleware function, which conflicts with proxy.ts.
Instead, custom JWT session management was implemented using jose (recommended
by the Next.js 16 authentication guide). This is simpler, has zero compatibility
issues, and is a better fit for a single-user personal tool.

Step 28. Install and configure Tailscale Funnel ✅

Prerequisites:

Tailscale installed on the host machine (already recommended in Phase 9)
Tailscale account with MagicDNS enabled
Funnel enabled on the tailnet (admin console → DNS → Enable HTTPS,
  then admin console → Access controls → add "funnel" attribute to the node)

Setup:

tailscale funnel 3000

This creates a public HTTPS endpoint at https://<hostname>.<tailnet>.ts.net
that proxies to localhost:3000. No client-side VPN or app needed — the phone
just opens the URL in a browser.

Make it persistent:

tailscale funnel --bg 3000

Or run as a system service so it survives reboots. On Windows, Tailscale
already runs as a service — the funnel flag persists automatically.

Next.js configuration:

The app receives requests through Tailscale's reverse proxy.
The 0.0.0.0 binding from Phase 9 already handles this — no additional config needed.

Deliverable:

eidetic is reachable from any device anywhere via a stable HTTPS URL.
No VPN app, no port forwarding, no dynamic DNS.

Step 29. Set up JWT session management ✅

Used:

jose — JWT signing and verification library (recommended by Next.js 16 docs)
  Works in all runtimes (Node.js, Edge, Proxy)

Installed:

npm install jose

Created:

lib/auth.ts — central auth module (server-only)
  hashPassword(password) — scrypt hash with random salt
  verifyPassword(password, stored) — timing-safe comparison
  isPasswordSet() — checks settings table for password_hash
  setPassword(password) — stores scrypt hash in settings table
  encrypt(payload) — signs a JWT with HS256
  decrypt(session) — verifies and decodes a JWT
  createSession() — creates a 30-day session cookie (eidetic-session)
  deleteSession() — removes the session cookie
  verifySession() — reads cookie, verifies JWT, checks expiry
  login(password) — validates against DB-stored hash, creates session
  logout() — deletes session and redirects to /login

lib/db.ts — added settings table (key TEXT PRIMARY KEY, value TEXT)
  Stores password_hash and future app configuration.

Environment:

.env.local (generated by install.ps1):
  AUTH_SECRET=<generated 32-byte hex secret>
  OLLAMA_MODEL=<chosen model based on VRAM>

Password is stored as a scrypt hash in the database, not in .env.local.
.env.local is in .gitignore (.env* pattern already present).

Deliverable:

JWT-based session management with httpOnly secure cookies.
Password stored securely in SQLite with scrypt hashing.

Phase 14 — Stalwart backbone for email + calendar (PLANNED)
═══════════════════════════════════════════════════════════════════════

Goal: bolt on the two data sources Eidetic doesn't yet understand — email
threads and the user's calendar — and let the LLM *manage* the calendar
(create / move / cancel events and tasks) via a standards-based server
instead of an ad-hoc cron table. Stalwart gives us IMAP + JMAP for mail,
CalDAV for calendar, CardDAV for contacts, and a single admin plane. All
data stays local; nothing moves to a third-party cloud.

Architectural shape this phase introduces:

  ┌──────────────────────┐            ┌──────────────────────────┐
  │    Stalwart          │   JMAP     │       Eidetic App        │
  │  SMTP / IMAP / JMAP  │◄──────────►│  email ingestion worker  │
  │  CalDAV / CardDAV    │   CalDAV   │  calendar sync worker    │
  │  single mailbox DB   │◄──────────►│  LLM calendar tool loop  │
  └─────────┬────────────┘            └────────────┬─────────────┘
            │                                      │
            │ fetched-accounts                     │ fragments +
            ▼ (pull from Outlook)                  ▼ wiki pages
  ┌──────────────────────┐            ┌──────────────────────────┐
  │  External provider   │            │  storage/eidetic.db +    │
  │  (Outlook / Gmail)   │            │  storage/chroma +        │
  └──────────────────────┘            │  storage/wiki            │
                                      └──────────────────────────┘

Key design decisions (to nail down before Step 46):

  1. Stalwart does NOT act as a public MX. Receiving mail directly means
     DNS, SPF/DKIM/DMARC, reverse DNS, and reputation work — all of
     which conflict with the "one laptop, one Funnel URL" install story.
     Stalwart will instead pull from the user's existing Outlook account
     via its fetched-accounts feature. Sending can go through the same
     provider via SMTP relay (or be deferred entirely — Phase 14 is
     read-and-manage, not send).
  2. Funnel only proxies HTTPS. JMAP and CalDAV are HTTPS → Funnel works.
     IMAP (143/993) and SMTP (25/465/587) are not reachable from outside
     the tailnet. That's fine: Eidetic talks to Stalwart over localhost,
     and phone clients can use JMAP-over-Funnel if ever needed.
  3. Calendar writes from the LLM go through CalDAV PUT, not a bespoke
     eidetic calendar table. The chain of custody is: chat → LLM emits
     tool call → Eidetic renders preview card → user confirms → CalDAV
     PUT → Stalwart → any standard client (iPhone Calendar, Thunderbird)
     sees the event. Eidetic's DB mirrors events only for RAG indexing.
  4. Single-user only. Stalwart supports multi-tenant; we won't. One
     mailbox, one calendar, one contact book. Eidetic's existing
     password gates Eidetic; Stalwart has its own admin credential
     stored alongside AUTH_SECRET in .env.local.

  library page shows them alongside uploaded files. RAG answers can
  cite email threads ("per your 2026-03-14 email with Lisa, the
  onboarding deck is due Friday").

Step 49. Email wiki pages — per-thread + per-correspondent

Two new wiki page types, produced by ingestion/wiki.py:

  thread     — one page per email thread. Summary + decisions + action
               items extracted by the LLM. Evidence section lists the
               individual message IDs.
  person     — one page per correspondent (email address). Aggregates
               all threads with that person; surfaces recurring topics
               and open action items. Analogous to the existing vendor
               pages for receipts.

Dirty marking (follows Phase 8 rules):

  new email in a thread → thread page for that thread is marked dirty,
                          person page for the sender is marked dirty
  new attachment parsed → the doc's own wiki page is dirty (existing)

Deliverable:

  A running synthesis of every ongoing conversation, queryable via the
  existing hybrid router. "What did I commit to Lisa about onboarding?"
  pulls the person page for Lisa + the RAG chunks from the actual
  emails.

Step 50. Calendar ingestion (read path)

Before the LLM can *manage* the calendar, Eidetic needs to *see* it.

New code:

  ingestion/calendar_sync.ts — Node worker (calendar work fits Node
    better than Python given the tool-calling loop will also be in TS)
    - uses tsdav (npm) to talk to Stalwart's CalDAV endpoint
    - on boot and every 5 minutes:
        fetch VEVENT + VTODO objects within a rolling window
        (past 30 days, future 180 days)
      parse via ical.js
    - upserts into a new events table:
        events (id, uid, calendar_id, summary, description, location,
                start_at, end_at, rrule, status, completed, etag,
                raw_ical, created_at, updated_at)
      etag is the CalDAV change token — drives incremental sync.
    - writes each event as a document_fragment with
        fragment_type = "event" (or "task" for VTODO)
      so events are first-class citizens in RAG retrieval. "When is
      my next meeting with David?" lands in the same retrieval path
      as any other question.

New API routes:

  GET  /api/calendar/events?from=...&to=...   — list events in range
  POST /api/calendar/sync                      — manual resync
  GET  /api/calendar/upcoming                  — next N events, used
                                                 by the home/chat UI

Schema migration: add events table and event-specific fragment columns
(event_uid, event_start_at, event_end_at) in the same lib/db.ts boot
migration pattern the rest of the app uses.

Deliverable:

  Every event in the user's calendar is visible in Eidetic, both as a
  live list and as RAG context. No writes yet.

Step 51. LLM-managed calendar — tool-calling loop

The core agentic step. The chat message route gains a tool-calling
dispatcher that recognizes calendar intents and lets the LLM act.

Intent detection in lib/queryRouter.ts:

  new category: "calendar_action"
  triggers on verbs: schedule, book, move, reschedule, cancel,
    remind me, add a task, mark done, add to calendar
  overrides rag/wiki/hybrid routing

Tool-calling transport:

  Gemma 3's tool-calling support via Ollama is partial. Two fallbacks:
    (a) structured-output mode — the system prompt forces a JSON
        response matching a toolCall schema; the server parses it and
        dispatches.
    (b) a small classifier LLM call first ("is this a calendar
        action? if yes, which tool + args in JSON?") then a normal
        reply LLM call framed around the tool result.
  Start with (a) since it's cheaper. Fall back to (b) if reliability
  is poor in practice.

Tools (lib/calendarTools.ts):

  listEvents(range: "today" | "this_week" | "next_week" | ISO range)
  createEvent(summary, start, end, description?, attendees?, location?)
  moveEvent(uid, newStart, newEnd)
  cancelEvent(uid)
  createTask(summary, due?, description?)
  completeTask(uid)

Each tool:
  - validates args with zod
  - for writes, builds a VEVENT/VTODO via ical.js
  - PUTs to the CalDAV collection (creates or updates)
  - refreshes the local events cache immediately so the response
    can cite the new/changed event

Deliverable:

  "Schedule lunch with David next Tuesday at 1pm" results in a CalDAV
  PUT against Stalwart, which iPhone Calendar sees within its next
  refresh. The LLM reply confirms with the event details and a link
  to the local event view.

Step 52. Preview + confirm UI for destructive actions

Per CLAUDE.md guidance: risky actions need explicit confirmation. The
calendar tool dispatcher runs in two modes:

  read-only tools (listEvents)          — execute immediately
  write tools (create/move/cancel/*)    — return a preview, not a commit

Preview flow in ChatView:

  - Assistant message contains a pending-action card (blue/yellow
    banner) instead of text: "Schedule 'Lunch with David' for
    2026-04-28 13:00-14:00? [Confirm] [Cancel]"
  - Confirm → POST /api/calendar/tools/execute with the tool payload.
    Route verifies the payload hasn't been tampered with (signed
    server-side when first proposed) and commits via CalDAV.
  - Cancel → discards the proposal; assistant posts a "cancelled"
    follow-up message.

sources_json column extended to carry pending tool payloads + their
signatures; survives reloads so a mid-flow refresh doesn't lose state.

Deliverable:

  The LLM never silently mutates the calendar. Every write passes
  through a human-in-the-loop confirmation with full context.

Step 54. Installer integration for Stalwart

Extend install.ps1 with an optional Stalwart setup block, matching
the pattern used for Tailscale Funnel in Phase 12:

  1. Print what Stalwart does and what it costs (a Windows service,
     ~200 MB RAM, storage/stalwart/ directory).
  2. Prompt: "Set up Stalwart for email + calendar? [y/N]"
  3. If yes:
     - Check if Stalwart is already installed (Get-Service "Stalwart").
     - Install via winget (preferred) or download-and-run the MSI
       from stalw.art/download if winget has no manifest yet.
     - Wait up to 60s for the admin port to come up.
     - Generate a random admin password + write to .env.local.
     - Provision the single mailbox + one calendar via the admin REST
       API.
     - Prompt for Outlook account details (app password or OAuth2
       device flow) and configure fetched-accounts.
  4. On failure: same try/catch/finally + Pause-Before-Exit treatment
     as the Tailscale block, with targeted error messages for the
     common fails (port 8080 in use, winget install blocked, Outlook
     OAuth denied).

launch.ps1 gains a matching verification step:

  - If STALWART_JMAP_URL is set in .env.local, check the service is
    running (Get-Service Stalwart) before starting the Eidetic server.
  - Print "Stalwart active" in green, or a yellow warning that
    email/calendar features will be inert until the service is up.

Deliverable:

  One install path, one launcher, Stalwart is a first-class optional
  component. Users who don't opt in still get the full Eidetic
  experience; users who do get email + calendar with no extra config.
