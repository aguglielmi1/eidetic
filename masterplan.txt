Phase 1 — Get a usable local chat stack running ✅ COMPLETE
Goal: prove the model, app shell, and local API all work before touching ingestion.
Step 1. Set up the local runtime ✅
Used:

Ollama (installed on Windows)
Gemma 4 (pulled via ollama pull gemma4)
Open WebUI was not used

Deliverable:
Model responds to API calls at http://localhost:11434/api/chat. Verified via PowerShell using Invoke-WebRequest.
Notes:

PowerShell aliases curl to Invoke-WebRequest, so standard curl syntax doesn't work. Use Invoke-WebRequest with -Method POST, -ContentType, and -Body parameters instead.
PowerShell may show a "Security Warning: Script Execution Risk" prompt when parsing web responses. Answering Y or adding -UseBasicParsing to the command avoids this in the future.
Gemma 4 was used instead of the originally planned Gemma 3. Update downstream references accordingly.

Phase 5 — Build the RAG layer ✅ COMPLETE

Goal: create the ground-truth retrieval system.

Step 9. Chunk the parsed fragments ✅

Approach: one chunk per fragment (no sub-chunking in v1).
Each fragment is already a meaningful unit:
  docs: one chunk per paragraph/heading/element
  slides: one chunk per text block or notes block
  spreadsheets: one chunk per row
  receipts: one chunk for raw OCR text, one for extracted fields

Deliverable:

chunks table in SQLite tracks which fragments have been embedded.
Each chunk is traceable back to its source fragment and document.

Step 10. Generate embeddings ✅

Used:

Ollama /api/embeddings endpoint
Model: nomic-embed-text (override with EMBED_MODEL env var)
Pre-requisite: ollama pull nomic-embed-text

Deliverable:

Every chunk has an embedding vector stored in Chroma.

Step 11. Store vectors and metadata ✅

Used:

chromadb PersistentClient at storage/chroma/
Collection: eidetic_chunks (cosine similarity space)

Stored per chunk:

chunk text (documents)
embedding vector (embeddings)
metadata: document_id, fragment_id, fragment_type,
          page/slide/sheet/row, vendor, receipt_date, total

SQLite chunks table mirrors Chroma IDs for integrity.

Implementation:

ingestion/embed.py — args: <doc_id> <db_path> [chroma_path] [ollama_url]
  Reads fragments → embeds each → upserts to Chroma + SQLite
  Re-embed deletes stale chunks first
  Updates documents.embed_status: null → embedding → embedded | embed_failed

POST /api/documents/[id]/embed — spawns embed.py detached

Deliverable:

Working semantic retrieval over all parsed data.

Each result includes:

chunk_id, text, score (cosine similarity 0–1)

Phase 7 — Build the hybrid query router ✅ COMPLETE

Goal: decide when to use raw RAG, wiki, or both.

Step 17. Create a retrieval orchestration API ✅

Used:

lib/queryRouter.ts — classification + retrieval orchestration
Modified app/api/conversations/[id]/messages/route.ts — inject context before Ollama call

Query categories implemented:

lookup           → RAG   (price, cost, date, how many, receipt, total, amount, invoice)
summary          → wiki first (summarize, overview, what is, tell me about, explain, how does)
comparison       → hybrid (compare, versus, vs, difference between, better than)
trend            → hybrid (trend, over time, historically, pattern, growth, decline)
citation_required→ RAG   (source, proof, evidence, cite, reference, verify)
chat             → no context (everything else — pure conversation)

Routing logic:

lookup / citation_required → RAG only
summary → wiki first; falls back to RAG if no wiki pages match
comparison / trend → hybrid (wiki + RAG together)
chat → passes message history to Ollama with no injected context

Deliverable:

lib/queryRouter.ts exposes routeQuery(query) which classifies the question,
runs retrieval, and returns { mode, category, systemPrompt, chunks, wikiPages, hasContext }.
The messages route prepends a typed system message when context was found.
X-Route-Mode header is returned on every chat response.

Step 18. Build hybrid context assembly ✅

For hybrid mode:

pull top wiki pages (up to 3, SQLite LIKE search on title + content)

send both to Ollama as a system message ahead of chat history
instruct model: raw sources are authoritative; wiki is synthesized memory; cite evidence; do not invent

System rules enforced:

raw source is authoritative
wiki is a convenience layer
cite raw evidence when possible
do not invent support

Context limits:

each wiki page capped at 1 500 chars in the system prompt

Deliverable:

Notes:

No new API routes added — orchestration lives inside the existing messages route.

Step 20. Mark affected wiki pages as dirty ✅

Dirty marking happens at three points:

re-parse (parse.py):  after successful parse, scans wiki_pages.source_doc_ids
                       for the parsed doc_id → sets dirty = 1 + enqueues rewiki job

re-embed (embed.py):  after successful embedding, same dirty marking logic

delete (DELETE /api/documents/[id]):  before deletion, marks wiki pages
                                       referencing the doc as dirty + enqueues rewiki job

Job deduplication: pending rewiki jobs for the same wiki page slug are not duplicated.

Examples:

new Staples receipt → vendor page “Staples” dirty
re-parsed onboarding deck → onboarding doc page dirty
deleted spreadsheet → monthly expense data page dirty

Deliverable:

wiki rebuilds become incremental instead of global.

Step 21. Rebuild only affected wiki pages ✅

Used:

ingestion/wiki.py `dirty` mode — rebuilds only wiki pages marked dirty
POST /api/wiki/rebuild-dirty — spawns wiki.py in dirty mode
GET /api/wiki/rebuild-dirty — returns dirty page count + pending jobs

Rebuild logic:

find all wiki_pages with dirty = 1
for each dirty page, gather valid source documents
if all sources deleted → remove orphaned wiki page + markdown file
otherwise → re-run process_document for the first source doc
  (vendor pages automatically aggregate all docs for that vendor)
job_queue entries updated: pending → running → completed | failed

Library UI:

amber banner shows count of out-of-date wiki pages
“Rebuild Dirty Pages” button triggers incremental rebuild
polls every 3s while rebuild is in progress
dirty count auto-clears when rebuild finishes

Wiki page index:

“Out of date” badge shown on dirty wiki pages in /wiki index

API routes added:

POST /api/wiki/rebuild-dirty — trigger dirty rebuild
GET  /api/wiki/rebuild-dirty — check dirty status + pending jobs

Notes:

wiki_pages.dirty is set to 0 by upsert_wiki_page (existing behavior from Phase 6).
Build verified clean.

Phase 11 — Secure remote access with Tailscale Funnel + auth ✅ COMPLETE

Goal: access eidetic from anywhere over HTTPS with no VPN client on the phone,
protected by real authentication so the Funnel URL is not an open door.

Implementation note:

The original plan specified NextAuth (Auth.js v5), but Next.js 16 renamed
middleware.ts to proxy.ts and deprecated the middleware convention. NextAuth v5
relies on wrapping the middleware function, which conflicts with proxy.ts.
Instead, custom JWT session management was implemented using jose (recommended
by the Next.js 16 authentication guide). This is simpler, has zero compatibility
issues, and is a better fit for a single-user personal tool.

Step 28. Install and configure Tailscale Funnel ✅

Prerequisites:

Tailscale installed on the host machine (already recommended in Phase 9)
Tailscale account with MagicDNS enabled
Funnel enabled on the tailnet (admin console → DNS → Enable HTTPS,
  then admin console → Access controls → add "funnel" attribute to the node)

Setup:

tailscale funnel 3000

This creates a public HTTPS endpoint at https://<hostname>.<tailnet>.ts.net
that proxies to localhost:3000. No client-side VPN or app needed — the phone
just opens the URL in a browser.

Make it persistent:

tailscale funnel --bg 3000

Or run as a system service so it survives reboots. On Windows, Tailscale
already runs as a service — the funnel flag persists automatically.

Next.js configuration:

The app receives requests through Tailscale's reverse proxy.
The 0.0.0.0 binding from Phase 9 already handles this — no additional config needed.

Deliverable:

eidetic is reachable from any device anywhere via a stable HTTPS URL.
No VPN app, no port forwarding, no dynamic DNS.

Step 29. Set up JWT session management ✅

Used:

jose — JWT signing and verification library (recommended by Next.js 16 docs)
  Works in all runtimes (Node.js, Edge, Proxy)

Installed:

npm install jose

Created:

lib/auth.ts — central auth module (server-only)
  hashPassword(password) — scrypt hash with random salt
  verifyPassword(password, stored) — timing-safe comparison
  isPasswordSet() — checks settings table for password_hash
  setPassword(password) — stores scrypt hash in settings table
  encrypt(payload) — signs a JWT with HS256
  decrypt(session) — verifies and decodes a JWT
  createSession() — creates a 30-day session cookie (eidetic-session)
  deleteSession() — removes the session cookie
  verifySession() — reads cookie, verifies JWT, checks expiry
  login(password) — validates against DB-stored hash, creates session
  logout() — deletes session and redirects to /login

lib/db.ts — added settings table (key TEXT PRIMARY KEY, value TEXT)
  Stores password_hash and future app configuration.

Environment:

.env.local (generated by install.ps1):
  AUTH_SECRET=<generated 32-byte hex secret>
  OLLAMA_MODEL=<chosen model based on VRAM>

Password is stored as a scrypt hash in the database, not in .env.local.
.env.local is in .gitignore (.env* pattern already present).

Deliverable:

JWT-based session management with httpOnly secure cookies.
Password stored securely in SQLite with scrypt hashing.

Step 37. Installer resilience and Windows compatibility ✅

The initial install.ps1 closed its window immediately on error when launched
via right-click → Run with PowerShell, making failures invisible. Several
Windows-specific incompatibilities also surfaced. All fixed:

Error visibility:
  - Wrapped the full script in try / catch / finally.
  - catch prints a red "Installation failed" banner with the exception message
    and InvocationInfo.PositionMessage (file + line + code fragment).
  - finally calls Pause-Before-Exit which waits for Enter before closing,
    so the window stays open whether install succeeds or fails.
  - Initialize $script:installFailed = $false before the try block so strict
    mode does not trip on the finally-block check when install succeeds.

Working directory:
  - Set-Location -Path $PSScriptRoot at the top of the try block.
  - Script now works regardless of where it was launched from (e.g. from
    C:\Users\<name> when invoked via an absolute path).
  - Fixes "npm error enoent ... package.json" when the caller's CWD
    wasn't the repo folder.

npm wrapper bug:
  - Node ships npm.ps1 which reads $MyInvocation.Statement; under
    Set-StrictMode -Version Latest this throws "The property 'Statement'
    cannot be found on this object".
  - Fix: call npm.cmd install and npm.cmd run build (the CMD shim,
    not the PowerShell wrapper). Avoids both the strict-mode bug and
    the default Windows execution policy blocking .ps1 scripts.

PowerShell 5.1 crypto compatibility:
  - [System.Security.Cryptography.RandomNumberGenerator]::Fill() is .NET Core
    only (PowerShell 7+). Fails on stock Windows PowerShell 5.1.
  - Fix: use ::Create().GetBytes($bytes) wrapped in try/finally with Dispose().
    Works on both Windows PowerShell 5.1 and PowerShell 7+.

Deliverable:

Installer runs reliably on a fresh Windows box, shows the real reason on
failure, and keeps the window open so the user can read it.

Step 40. Incremental launch script ✅

launch.ps1 complements install.ps1. Run it after cloning/installing once;
it pulls changes and only re-syncs what actually moved:

1. Pull updates
   - git status --porcelain first: if the tree is dirty, skip the pull
     and print the dirty list (never stomp on in-progress work).
   - Otherwise git fetch + compare HEAD to @{u}. If ahead, git pull --ff-only.
   - Before pulling, hash package.json, package-lock.json, and
     ingestion/requirements.txt. After pulling, re-hash and record which
     files changed — used to drive the next steps.

2. Node dependencies
   - npm.cmd install only if node_modules is missing OR the manifest
     changed during pull. Otherwise skip with "node_modules in sync
     with lockfile".

3. npm audit
   - npm.cmd audit --json, parsed for metadata.vulnerabilities.total.
   - If zero → skip. If >0 → npm.cmd audit fix. If parse fails → run
     audit fix anyway (defensive).

4. Python dependencies
   - pip install -r ingestion/requirements.txt only when that file
     actually changed during pull.

5. Build
   - npm.cmd run build only if updates were pulled OR .next is missing.
   - Otherwise skip with ".next is current".

6. Start
   - Prints local + public URL banner, then npm.cmd start (blocks).

Step 41. Launcher window-visibility fix ✅

launch.ps1 closed its PowerShell window too fast to read when launched
via right-click → Run with PowerShell. Two root causes, two fixes:

PowerShell 5.1 parse errors from Unicode in comments:
  - The script used box-drawing ── and em-dash — in section comments.
  - Windows PowerShell 5.1 reads UTF-8 files without a BOM as Windows-1252;
    the multi-byte UTF-8 sequences decoded to characters that tripped the
    parser with cascading "Missing terminator" / "Missing catch or finally"
    errors — all stemming from a single mangled byte.
  - Fix: replaced every ── and — in launch.ps1 with ASCII (---, -).
    Script now parses identically regardless of how PS reads encoding.

launch.cmd wrapper:
  - Small .cmd shim that invokes
      powershell.exe -NoProfile -ExecutionPolicy Bypass -File launch.ps1
    and runs `pause` unconditionally at the end.
  - Keeps the window open even when launch.ps1 fails before its own
    Pause-Before-Exit runs (e.g. execution-policy rejection, or the parse
    errors above).
  - Users should double-click launch.cmd, not launch.ps1.
